socket.io aws elasticbeanstalk + nginx 환경에서 사용하기

발단

기존에 존재하는 소켓 서버가 한대가 있고, 소켓 서버를 이용하는 application 서버가 여러대 있었다.
소켓 서버로 트래픽이 몰리게 되면 Auto scaling , load balancer 달려있지 않아서 대비책은 서버 사양을 올리는 방법 뿐이였다.
이러한 방법은 한계가 있기 때문에. 소켓 서버 비용도 절감하면서 다중 서버 환경을 만들기 위해 진행했던 방법이다.

소켓 서버에 트래픽이 몰리게 되는 현상은 application 서버에 요청이 늘어나 Auto scaling 의해 여러대로 늘어난 상황이다.
application 서버에 남는 자원을 활용해 소켓 서버를 application 서버에 합쳐 같이 Auto scaling 의해 관리되게 변경하게 되었다.
소켓 서버 비용도 들지않고, 단일 서버 문제도 해결할 수 있었다.

application load balancer 으로 변경하면 간단하게 해결될 일이지만 가능한 환경을 바꾸지 않기 위해 기존에 존재하는
eb설정을 변경하지 않고 진행하고자 생각했다. 서버는 aws eb를 사용하고 classic load balancer 사용하기 때문에 프록시에 문제가 있었는데 아래 방법으로 해결하였다.

nginx + socket.io 사용하기

eb를 사용해서 express + socket.io 사용할 때 aws console 에서 eb를 만들게 되면
classic load balancer 사용하는데 해당 load balancer는 proxy 기능이 없어서 곤란할 때가 있었습니다.
이때 nginx으로 해결 할 수 있는 방법이 있습니다.

eb ebextensions에 nginx 환경 설정 파일 생성

  • upstream 에서 nodejs, socket 나뉘어 지는데 들어오는 각각 서버에 대한 포트번호 입니다.
  • proxy_set_header connection upgrade 해야 client 에서 핸드쉐이킹 오류가 발생하지 않습니다.

nginx를 사용한 socket.io에 연결하는 client 설정

  • nginx를 사용해서 socket.io 에 연결할때 client socket.io.resource에서 nginx socket.io 설정한 location으로 지정해줘야 합니다.

  • ‘nginx path’/socket.io 같은 형식입니다.

socket.io 다중 서버 환경에서 session id unknown 발생 할 때

다중 서버 환경에서 nginx + socket.io 사용하는 환경에서 발생하기도 합니다.
A 서버로 소켓 연결을 시도하면 A서버에 롱 폴링으로 인증이 완료된 뒤 폴링을 끊고 A서버로 소켓 연결을 해야하는데 B서버로 롱폴링 연결을 해버리면 socket.io 에서 session id unknown 오류가 발생합니다. 이때 해결 방안은 socket.io client 에서 웹소켓을 강제 사용으로 설정하면 해결 할 수 있습니다.
하지만 강제로 사용하면 브라우저가 웹소켓을 지원하지 않을때 문제가 발생하게 되는데, 강제로 지정하지 않고 해결할 수 있는 방법은 elb sticky session 이용하면 됩니다.
